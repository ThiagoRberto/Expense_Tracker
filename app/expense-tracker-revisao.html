Princípio central: A camada de serviço (financial_service.py) é 100% IO-Free — não lê banco, não lê relógio, não acessa estado global. Todas as funções são puras e recebem dataclasses como parâmetros. Isso é o que torna os testes unitários possíveis sem qualquer mock.

Tratamento de erros global

Em main.py, qualquer ValueError disparado em qualquer lugar retorna HTTP 422:

@app.exception_handler(ValueError)
async def value_error_handler(request, exc):
    return JSONResponse(status_code=422, content={"detail": str(exc)})

Outros códigos de erro usados: 404 (usuário não encontrado), 409 (conflito/unique constraint), 422 (validação Pydantic ou ValueError).

Fluxo de uma Requisição
~5 min

Exemplo: GET /users/3/summary → analytics.py

1
HTTP Request chega ao Router
routers/analytics.py FastAPI roteia para get_user_summary(user_id=3). O parâmetro user_id vem da URL.
2
Dependency Injection — Sessão + 404
dependencies.py get_user_or_404(3, db, with_relations=True) faz uma única query com joinedload em bills/expenses/incomes/investments (evita N+1). Se não achar → HTTP 404.
3
Conversão ORM → Dataclasses puras
routers/analytics.py Cada ORM object é convertido: IncomeData(i.income_value), _to_expense_data(e), etc. A partir daqui, zero SQL.
4
Service Layer (puro)
services/financial_service.py summarize_finances(incomes, expenses, bills, investments, budget_ceiling) chama internamente calculate_balance, calculate_net_worth e, se budget_ceiling != None, check_budget_alert. Retorna um dict.
5
Schema de resposta (Pydantic)
schemas/summary.py FastAPI serializa o dict retornado via FinancialSummary (response_model). Campos extras são descartados automaticamente.
6
JSON Response → Cliente
FastAPI serializa para JSON e responde HTTP 200.

Onde o "agora" (relógio) é injetado

O serviço nunca lê o relógio. O datetime.now() é chamado no router (analytics.py:113) e o resultado é passado como parâmetro para project_monthly_invoices. Isso mantém a função de serviço determinística e testável sem mocks de tempo.

Service Layer — Todas as Funções
~14 min
calculate_balance
(incomes, expenses, bills) → float
saldo = Σ income_value − Σ (expense_value / max(installment, 1)) − Σ bill_value
Regras
  • Despesa parcelada contribui apenas com 1 parcela por mês
  • installment=0 é tratado como 1 (evita divisão por zero) via max(installment, 1)
  • Resultado negativo = déficit mensal — comportamento correto
  • Resultado arredondado em 2 casas
Validações (→ ValueError)
  • income_value < 0
  • expense_value < 0
  • bill_value < 0
Atenção: installment=0 usa política tolerante (max(installment,1)). Já project_installments usa política estrita (levanta ValueError). As duas são intencionais — conhecer a diferença é importante.
check_budget_alert
(category_total, budget_ceiling) → BudgetStatus
ratio = category_total / budget_ceiling
ratio ≤ 0.80 OK
0.80 < ratio < 1.0 WARNING
ratio ≥ 1.0 EXCEEDED
Fronteira crítica: ratio = 0.8 exato é OK (condição é ratio > 0.8, não >=). O teste confirma isso.
MC/DC (2 condições)
#C1 ≥1.0C2 >0.8Res
1FFOK
2FTWARN
3TTEXC
Validações (→ ValueError)
  • budget_ceiling <= 0
  • category_total < 0
project_installments
(expense_value, installments, start_month, start_year) → list[InstallmentEntry]
monthly = round(expense_value / installments, 2)
last_amount = round(expense_value − monthly × (installments − 1), 2) ← absorve arredondamento
Comportamento chave
  • Virada de ano: month > 12 → month=1, year+=1
  • Última parcela absorve o erro de arredondamento (10/3 → 3.33, 3.33, 3.34)
  • Política estrita: installments <= 0 levanta ValueError
Validações (→ ValueError)
  • expense_value < 0
  • installments <= 0
  • start_month fora de [1, 12]
  • start_year < 1
# exemplo: 10.0 em 3x a partir de jan/2025
project_installments(10.0, 3, 1, 2025)
# → [InstallmentEntry(1, 1, 2025, 3.33),
#    InstallmentEntry(2, 2, 2025, 3.33),
#    InstallmentEntry(3, 3, 2025, 3.34)]  ← último absorve diferença
project_monthly_invoices
(expenses, reference_month, reference_year, months_ahead) → list[MonthlyInvoice]

Consolida parcelas de todas as despesas em faturas mensais, respondendo "quanto vou dever em cada mês futuro".

Algoritmo interno
  1. Calcula índice absoluto: _to_month_index(month, year) = year*12 + (month−1)
  2. Cria janela de meses: range(start_index, start_index + months_ahead)
  3. Para cada despesa, chama project_installments
  4. Soma parcelas que caem dentro da janela
  5. Converte índices de volta via _from_month_index
Garantias
  • Todo mês da janela aparece no resultado, mesmo com total=0.0
  • Parcelas passadas (antes da referência) são ignoradas
  • Parcelas além da janela são ignoradas
  • Suporta virada de ano via índice absoluto
  • Herda validação estrita de project_installments
# 300 em 3x a partir de nov/2025, janela de 3 meses
project_monthly_invoices([ExpenseData(300, 3, 11, 2025)], 11, 2025, 3)
# → [MonthlyInvoice(11, 2025, 100.0),
#    MonthlyInvoice(12, 2025, 100.0),
#    MonthlyInvoice(1,  2026, 100.0)]  ← virada de ano
Validações (→ ValueError)
  • reference_month fora de [1, 12]
  • reference_year < 1
  • months_ahead < 1
  • Herda: installments <= 0
MC/DC reference_month
#C1 <1C2 >12Res
1FFválido
2FTraises
3TFraises
Caso 4 (T,T) = inexecutável
calculate_net_worth
(investments, balance) → float
patrimônio = balance + Σ (value_invested + dividends)

balance aqui é o saldo mensal retornado por calculate_balance. Saldo negativo reduz o patrimônio — comportamento correto. Lista vazia de investimentos retorna apenas o saldo.

summarize_finances
(incomes, expenses, bills, investments, budget_ceiling=None) → dict

Função de composição — chama calculate_balance e calculate_net_worth internamente.

{
  "balance":        float  # saldo mensal
  "net_worth":      float  # patrimônio líquido
  "total_income":   float  # soma das receitas
  "total_spending": float  # despesas parcela + contas fixas
  "budget_status":  "OK"|"WARNING"|"EXCEEDED"  ← só se budget_ceiling != None
}
calculate_category_totals
(expenses) → dict[str, float]

Agrupa o custo mensal (fração da parcela) por categoria. Cada despesa contribui com expense_value / max(installment, 1).

expenses = [
    ExpenseData(600, 2, 1, 2025, "alimentação"),  # 300/mês
    ExpenseData(300, 1, 1, 2025, "alimentação"),  # 300/mês
    ExpenseData(400, 1, 1, 2025, "transporte"),   # 400/mês
]
calculate_category_totals(expenses)
# → {"alimentação": 600.0, "transporte": 400.0}

Categorias sem teto definido não aparecem em check_all_category_alerts.

check_all_category_alerts
(category_totals, category_budgets) → dict[str, BudgetStatus]

Aplica check_budget_alert para cada CategoryBudgetData.

Regras de cobertura
  • Categoria com teto mas sem despesas → total=0.0 → OK
  • Categoria com despesas mas sem teto → ignorada
Validação (→ ValueError)
  • Qualquer ceiling <= 0 levanta ValueError

Checklist de Revisão Rápida
~3 min
  • A camada de serviço é IO-Free: nenhuma função acessa banco, arquivo ou relógio
  • calculate_balance: usa fração mensal da despesa (value/max(installment,1)), política tolerante
  • project_installments: política estrita (installments <= 0 → ValueError), última parcela corrige arredondamento
  • check_budget_alert: fronteira em 0.8 é exclusiva (ratio > 0.8), ou seja, 80% exato ainda é OK
  • project_monthly_invoices: todos os meses da janela aparecem mesmo com total=0.0; parcelas fora da janela são ignoradas
  • summarize_finances: budget_status só aparece no dict se budget_ceiling != None
  • check_all_category_alerts: categoria com teto mas sem despesas = OK; categoria sem teto = ignorada
  • O datetime.now() é chamado no router, não no serviço — mantém pureza e testabilidade
  • get_user_or_404 com with_relations=True usa joinedload para evitar N+1 queries
  • ValueError em qualquer lugar → HTTP 422 (handler global em main.py)
  • Técnicas de teste usadas: MC/DC, BVA (Boundary Value Analysis), caminho feliz, edge cases, validações de entrada
  • Testes unitários: 0 mocks, só dataclasses + assert. Nenhuma dependência de banco ou sessão
